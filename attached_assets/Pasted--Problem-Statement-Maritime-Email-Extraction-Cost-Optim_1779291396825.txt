
Problem Statement: Maritime Email Extraction & Cost Optimization System
Project Background
In the shipbroking and chartering industry, brokers and charterers receive thousands of emails daily containing cargo requirements, vessel availability, voyage charter details, and time charter information.
These emails are generally unstructured and contain operational details in multiple formats, abbreviations, and free-text maritime terminology.
Currently, we use Large Language Model (LLM) APIs to extract structured information from these emails. While extraction accuracy is acceptable, the operational cost is extremely high due to:
Large email volume
Long email content
Multiple extraction calls per email
Continuous daily processing
Repeated parsing of similar formats
We want to significantly reduce or eliminate dependency on expensive LLM APIs while maintaining high extraction accuracy.
The goal is to design and implement a scalable, intelligent, low-cost extraction architecture.

Existing Challenges
1. Extremely High LLM API Cost
Current extraction heavily relies on external LLM APIs.
Problems:
High token consumption
Long maritime emails
Multiple extraction retries
Thousands of emails daily
High recurring monthly cost

2. Repetitive Email Patterns
Most maritime emails follow repetitive formats:
Fixed broker templates
Standard abbreviations
Predictable field locations
Common voyage terminology
Repeated vessel descriptions
Using expensive LLM inference for every email is inefficient.

3. Multi-Entry Parsing Complexity
One email may contain:
Multiple cargoes
Multiple vessels
Mixed VC and TC requirements
Signature blocks
Operational clauses
System must split and extract all entries correctly.

Objective
Design a hybrid extraction architecture that:
Reduces LLM API usage by 70–95%
Maintains high extraction accuracy
Supports maritime abbreviations and formats
Handles bulk email processing
Is scalable and production-ready

Expected Solution Approach
The developer is expected to propose and implement a cost-optimized architecture.
Possible approaches may include:

1. Rule-Based Extraction Layer
Use:
Regex
Pattern matching
Maritime dictionaries
NLP tokenization
Template detection
Examples:
LAYCAN:
DWT:
LP:

REDEL:

This layer should extract straightforward fields without LLM calls.

2. Template Classification Engine
Detect recurring broker templates and process them using predefined parsers.
Example:
Broker A format
Broker B format
SeaSchiffe format
YB Global Shipping format

Goal:
Avoid LLM calls for known formats
Improve speed
Reduce cost

3. Lightweight ML / Local NLP Models
Possible options:
spaCy NER
DistilBERT
Small local transformer models
Fine-tuned maritime extraction model
Goal:
Run locally
Avoid per-token API cost
Improve entity extraction

4. Hybrid Confidence-Based Pipeline
Suggested flow:
Step 1:
Rule Engine

Step 2:
Template Parser

Step 3:
Local ML Model

Step 4:
Fallback to LLM only if confidence is low

This ensures LLM is used only for difficult emails.

5. Maritime Knowledge Base
Create dictionaries for:
Ports
Regions
Cargo names
Vessel types
Maritime abbreviations
Country mapping
Example:
ECI = East Coast India
WAFR = West Africa
UMX = Ultramax
SMX = Supramax


6. Signature Parsing Engine
Extract:
PIC
Email
Phone Number
from email signatures separately using regex and heuristics.

7. Email Segmentation Engine
Split emails into:
Cargo blocks
Vessel blocks
Signature blocks
Operational notes
before extraction.
